The automation engineer's mindset: how to identify, prioritise, and build automation that sticks

Amal Jose20246 min readCloud Operations & Automation Engineer

After 11 years of building automation across cloud platforms, provisioning systems, CI/CD pipelines, and operational tooling, the most important thing I have learned is this: not every manual process deserves a script. The engineers who automate everything indiscriminately create a different kind of toil — maintaining brittle automation, debugging scripts that nobody remembers writing, and reverting automation that made a process faster but less safe.

The automation engineer's real skill is not Python or Bash. It is knowing what to automate, when, how to scope it correctly, and how to build it in a way that does not rot six months later when the process it automates changes.

Step 1: identify what actually deserves automation

The classic SRE framework asks: how many hours per week does this manual task consume? If the answer is under half an hour, automate it only if the risk of human error is high. If the answer is above two hours, automate it regardless of complexity. Everything in between requires a judgment call.

But frequency and time are not the only dimensions. I use four criteria:

CriterionQuestions to askWeight
FrequencyHow often does this happen? Daily? Weekly? Monthly?High
Error riskWhat happens when a human makes a mistake here? Is it recoverable?High
Toil intensityIs this creative work or mechanical repetition? Would a good engineer learn anything doing this the 50th time?Medium
Process stabilityHow often does the underlying process change? Automating something that changes monthly is expensive maintenance.Medium

The highest-priority automation candidates score high on frequency and error risk simultaneously. Certificate renewals: frequent enough (quarterly), high error risk (expiry takes down production), stable process (the CA API does not change). This is why certificate lifecycle automation is one of the first things I build on any new platform.

The lowest-priority candidates are low-frequency with low error risk. A quarterly report that takes 20 minutes to compile manually is not worth 40 hours of engineering to automate — unless the business is growing and "quarterly" becomes "weekly."

Step 2: scope it correctly before writing a line of code

Most automation projects fail not in the implementation but in the scoping phase. The engineer starts automating one thing, discovers two adjacent things that "might as well" be automated while they are at it, and six weeks later has a half-finished system that automates nothing completely.

Before writing code, answer these questions in writing:

  • What is the exact start and end state? "The process is automated" is not a success criterion. "A Slack message is sent with a migration summary, all artifacts are validated, and the ticket is closed" is a success criterion.
  • What does failure look like and what should happen? Every automation that can fail without the failure being immediately obvious is dangerous automation. Define failure modes explicitly and build detection and alerting for them.
  • What are the human approval gates? Not everything should be automated end-to-end. Some processes have approval steps that exist for good reasons — compliance, risk management, a second pair of eyes before an irreversible action. Build the automation around those gates, not over them.
  • How will this be maintained when the process changes? Who owns it? Where does the code live? How is it tested?

The scope trap: "While I am at it" are the four most expensive words in automation engineering. Scope every automation project to one complete, valuable thing. Ship that. Then decide whether to extend it.

Step 3: build for operability, not just functionality

A script that works on your laptop is not automation. Automation that runs reliably in production, surfaces failures clearly, and can be debugged by someone other than the original author — that is automation.

Idempotency is non-negotiable

Every automation step should be safe to re-run. If the pipeline fails halfway through and someone reruns it, the parts that already completed should not be re-executed in a way that causes errors or data duplication. This sounds obvious and is routinely ignored.

def create_resource_idempotent(client, name, config):
    """Create a resource only if it doesn't already exist."""
    existing = client.get(name)
    if existing:
        # Already exists — check if it matches what we want
        if matches_desired_state(existing, config):
            logger.info(f"{name} already in desired state, skipping")
            return existing
        else:
            logger.info(f"{name} exists but differs, updating")
            return client.update(name, config)
    else:
        logger.info(f"Creating {name}")
        return client.create(name, config)

Structured logging in automation code

When automation runs at 3am and something goes wrong, the log is your only debugging interface. Log at the right level, log structured data, and log enough context to reconstruct what happened without SSH access to the runner.

import logging, json
from datetime import datetime, timezone

class StructuredLogger:
    def __init__(self, name, context=None):
        self.logger = logging.getLogger(name)
        self.context = context or {}

    def info(self, msg, **kwargs):
        self._log('INFO', msg, **kwargs)

    def error(self, msg, **kwargs):
        self._log('ERROR', msg, **kwargs)

    def _log(self, level, msg, **kwargs):
        record = {
            'timestamp': datetime.now(timezone.utc).isoformat(),
            'level': level,
            'message': msg,
            **self.context,
            **kwargs
        }
        print(json.dumps(record))   # structured output, Splunk/Loki friendly

# Usage
log = StructuredLogger('cert-pipeline', context={
    'run_id': 'abc-123',
    'landscape': 'prod',
    'component': 'cert-renewal'
})

log.info('Starting renewal', cert_alias='api-gateway-prod', days_remaining=12)
log.error('Renewal failed', cert_alias='api-gateway-prod', error='CA timeout', retry=True)

Dry-run mode

Any automation that mutates state should have a --dry-run flag that shows exactly what it would do without doing it. This is the single feature that makes operations teams trust automation — they can verify the behaviour before the first production run.

def deploy_certificate(target, cert_pem, key_pem, dry_run=False):
    logger.info("Deploying certificate", target=target, dry_run=dry_run)
    if dry_run:
        logger.info("DRY RUN: would deploy cert to target — no changes made",
                    target=target,
                    cert_expiry=get_expiry(cert_pem).isoformat())
        return {"status": "dry_run", "target": target}
    # Actual deployment
    result = _do_deploy(target, cert_pem, key_pem)
    logger.info("Certificate deployed", target=target, status=result["status"])
    return result

Step 4: maintain it like production code

Automation that is not maintained becomes the thing it was meant to eliminate — a source of toil and risk. The practices that keep automation healthy over time:

Test your automation

Unit tests for the logic. Integration tests against a sandbox environment. At minimum, a test that runs the automation in dry-run mode against a real (non-production) target on every pull request. Untested automation is untrustworthy automation.

Version and deploy it like any other service

Automation code lives in Git. Changes go through PR review. Deployments are tracked. If you cannot answer "what version of the provisioning script ran last Tuesday?", you cannot debug last Tuesday's incident.

Set a review cadence

Every six months, review each automation and ask: does the process it automates still work this way? Are the APIs it calls still current? Has the error handling kept up with new failure modes? Schedule this review as a recurring calendar event, not a "someday" task.

The meta-skill: knowing when to stop

The hardest automation decision is not what to build — it is knowing when to stop. I have seen teams spend 200 hours automating a process that takes 5 hours per year manually, justifying it as "principle." I have seen automation built to eliminate a human approval step that existed for a reason, resulting in an incident that cost more than the toil it saved.

Automation is a tool in service of reliability and engineering throughput. When it serves those goals, build it. When it does not — when the process is too unstable, too infrequent, or too important to remove humans from — acknowledge that and invest the time elsewhere.

The automation engineer who can make that call is more valuable than the one who can write the fastest script.
