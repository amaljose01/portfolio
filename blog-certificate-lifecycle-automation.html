Automating certificate lifecycle end-to-end: from discovery to rotation with Python and Jenkins

Amal Jose20257 min readCloud Operations & Automation Engineer

Certificate expiry incidents are one of the most embarrassing categories of production outage. They are not caused by bugs, complex infrastructure failures, or unexpected load — they are caused by a date on a calendar that everyone knew about and nobody acted on in time. I have seen APIs go dark, mutual TLS handshakes fail, and customers locked out — all because a cert expired at midnight and there was no automated process to catch it.

This post documents the end-to-end certificate lifecycle automation pipeline I built, covering discovery, triage, renewal via a CA API, and deployment — with secrets handled properly throughout.

Where certificates live (and get forgotten)

In any reasonably complex cloud environment, certificates appear across multiple systems:

  • Kubernetes secrets — TLS certs for ingress controllers and service-to-service mTLS
  • Load balancer listeners — SSL termination certs on ALBs, NLBs, or nginx
  • Application keystores — JKS or PKCS12 files embedded in application config
  • API gateway configurations — client certs and backend mTLS certs
  • Secret managers — Vault, AWS Secrets Manager, Azure Key Vault

Each has a different API, a different renewal process, and usually a different owner. Before automation, the standard approach was a spreadsheet updated quarterly — which is not frequent enough for 90-day certificates.

Pipeline architecture

The automation runs as a Jenkins pipeline on a daily schedule with four stages:

  1. Discovery — scan all certificate stores, collect expiry metadata
  2. Triage — classify by urgency: expired, critical (<14 days), warning (<30 days), healthy
  3. Renewal — for critical/warning certs, trigger renewal via CA API
  4. Deployment — push renewed certs to targets, validate, update secret stores

Stage 1: Discovery

The discovery module queries each certificate source and normalises results into a common data model:

from dataclasses import dataclass
from datetime import datetime, timezone

@dataclass
class CertRecord:
    alias: str
    source: str          # 'k8s', 'vault', 'aws_acm', 'file'
    common_name: str
    expiry: datetime
    days_until_expiry: int
    status: str          # 'expired' | 'critical' | 'warning' | 'healthy'
    auto_renewable: bool
    renewal_owner: str

def classify(days: int) -> str:
    if days <= 0:  return 'expired'
    if days <= 14: return 'critical'
    if days <= 30: return 'warning'
    return 'healthy'

Discovering certs from Kubernetes

from kubernetes import client, config
from cryptography import x509
from cryptography.hazmat.primitives.serialization import Encoding
import base64

def discover_k8s_certs(namespace='*') -> list[CertRecord]:
    config.load_incluster_config()
    v1 = client.CoreV1Api()
    secrets = v1.list_secret_for_all_namespaces(field_selector='type=kubernetes.io/tls')
    records = []
    for s in secrets.items:
        tls_crt = base64.b64decode(s.data['tls.crt'])
        cert = x509.load_pem_x509_certificate(tls_crt)
        expiry = cert.not_valid_after_utc
        days = (expiry - datetime.now(timezone.utc)).days
        records.append(CertRecord(
            alias=f"{s.metadata.namespace}/{s.metadata.name}",
            source='k8s',
            common_name=cert.subject.get_attributes_for_oid(
                x509.NameOID.COMMON_NAME)[0].value,
            expiry=expiry,
            days_until_expiry=days,
            status=classify(days),
            auto_renewable=True,
            renewal_owner='platform-ops'
        ))
    return records

Stage 2: Triage and notification

After discovery, the pipeline routes notifications to the right owners — not everyone for everything:

  • Expired or critical → immediate Slack alert to on-call channel + auto-create JIRA ticket
  • Warning → daily digest to the ops channel
  • Healthy → no notification (silence is signal-free)
  • Auto-renewable + platform-owned → skip notification entirely, proceed directly to renewal

Stage 3: Renewal via CA API

For publicly trusted certificates, renewal goes through your CA's API (Digicert, Sectigo, Let's Encrypt via ACME). The pattern is the same regardless of CA: generate a fresh key pair and CSR, submit to the CA, poll until issued.

from cryptography.hazmat.primitives.asymmetric import rsa
from cryptography.hazmat.primitives import hashes, serialization
from cryptography import x509
from cryptography.x509.oid import NameOID

def generate_key_and_csr(common_name: str, sans: list[str]):
    """Always generate a fresh key — never reuse the old one."""
    key = rsa.generate_private_key(public_exponent=65537, key_size=2048)
    csr = (
        x509.CertificateSigningRequestBuilder()
        .subject_name(x509.Name([
            x509.NameAttribute(NameOID.COMMON_NAME, common_name),
            x509.NameAttribute(NameOID.ORGANIZATION_NAME, "Your Org"),
        ]))
        .add_extension(
            x509.SubjectAlternativeName([x509.DNSName(n) for n in sans]),
            critical=False,
        )
        .sign(key, hashes.SHA256())
    )
    key_pem = key.private_bytes(
        serialization.Encoding.PEM,
        serialization.PrivateFormat.TraditionalOpenSSL,
        serialization.NoEncryption()
    )
    csr_pem = csr.public_bytes(serialization.Encoding.PEM)
    return key_pem, csr_pem

def request_certificate(ca_api_url, api_key, csr_pem, common_name):
    import requests
    resp = requests.post(
        f"{ca_api_url}/certificates",
        headers={"X-API-Key": api_key},
        json={"csr": csr_pem.decode(), "validity_days": 365, "cn": common_name}
    )
    resp.raise_for_status()
    return resp.json()["order_id"]

def poll_until_issued(ca_api_url, api_key, order_id, timeout=3600):
    import requests, time
    headers = {"X-API-Key": api_key}
    start = time.time()
    while time.time() - start < timeout:
        r = requests.get(f"{ca_api_url}/orders/{order_id}", headers=headers).json()
        if r["status"] == "issued":
            return r["certificate_pem"]
        if r["status"] == "failed":
            raise Exception(f"CA rejected order {order_id}: {r.get('reason')}")
        time.sleep(60)
    raise TimeoutError(f"Order {order_id} not issued within {timeout}s")

Security rule: Always generate a fresh private key on every renewal. Never reuse the existing key — if it was compromised during the certificate's lifetime, reusing it negates the entire point of renewal.

Stage 4: Deployment

Kubernetes secrets

def deploy_to_k8s(namespace, secret_name, cert_pem: bytes, key_pem: bytes):
    from kubernetes import client, config
    import base64
    config.load_incluster_config()
    v1 = client.CoreV1Api()
    secret_body = client.V1Secret(
        metadata=client.V1ObjectMeta(name=secret_name, namespace=namespace),
        type='kubernetes.io/tls',
        data={
            'tls.crt': base64.b64encode(cert_pem).decode(),
            'tls.key': base64.b64encode(key_pem).decode(),
        }
    )
    try:
        v1.replace_namespaced_secret(secret_name, namespace, secret_body)
    except Exception:
        v1.create_namespaced_secret(namespace, secret_body)

    # Store private key in Vault — never keep it in memory longer than needed
    store_key_in_vault(f"certs/{namespace}/{secret_name}/key", key_pem)
    key_pem = None  # clear reference

Post-deployment validation

import ssl, socket
from cryptography import x509 as cx509

def validate_deployed_cert(hostname: str, port: int, expected_serial: int) -> bool:
    ctx = ssl.create_default_context()
    with ctx.wrap_socket(socket.socket(), server_hostname=hostname) as s:
        s.connect((hostname, port))
        der = s.getpeercert(binary_form=True)
    live_cert = cx509.load_der_x509_certificate(der)
    return live_cert.serial_number == expected_serial

The Jenkinsfile

pipeline {
  agent any
  triggers { cron('H 6 * * *') }  // run daily at 6am
  stages {
    stage('Discover') {
      steps {
        withCredentials([string(credentialsId: 'vault-token', variable: 'VAULT_TOKEN')]) {
          sh 'python3 cert_pipeline.py discover --output inventory.json'
        }
      }
    }
    stage('Triage') {
      steps {
        sh 'python3 cert_pipeline.py triage --input inventory.json --output triage.json'
        archiveArtifacts 'triage.json'
      }
    }
    stage('Renew') {
      steps {
        withCredentials([string(credentialsId: 'ca-api-key', variable: 'CA_API_KEY')]) {
          sh 'python3 cert_pipeline.py renew --input triage.json'
        }
      }
    }
    stage('Deploy & Validate') {
      steps {
        sh 'python3 cert_pipeline.py deploy --validate'
      }
      post {
        failure {
          slackSend channel: '#ops-alerts',
            color: 'danger',
            message: "Certificate deployment FAILED — manual intervention required"
        }
      }
    }
  }
}

Results

Since deploying this pipeline, there have been zero certificate expiry incidents. Certificates are renewed automatically 30 days before expiry. The operations team's involvement is limited to approving change requests for certs that require security sign-off. What was previously a quarterly manual audit that occasionally missed things is now a fully auditable, daily automated process with a clear trail from discovery through renewal through deployment.

If your team is still tracking certificates in a spreadsheet — or worse, relying on Outlook reminders — this is one of the highest-return automation investments you can make. The blast radius of a missed cert is high; the cost of building the automation is low.
