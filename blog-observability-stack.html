Building a multi-layer observability stack: Prometheus, Grafana, and Splunk without the noise

Amal Jose20259 min readCloud Operations & Automation Engineer

Every platform team I have worked on eventually arrives at the same uncomfortable realisation: having monitoring tools is not the same as having observability. You can have Prometheus, Grafana, and Splunk all running simultaneously and still find yourself SSH-ing into a server at 3am trying to understand what is happening — because the tools are generating noise, not signal.

This post is about how I have structured multi-tool observability stacks to actually work in production: clear ownership boundaries between tools, alert rules that mean something, dashboards engineers actually open during incidents, and a log pipeline that does not cost more than the application it monitors.

The problem: tools without a strategy

The typical evolution looks like this. A team starts with basic server metrics — CPU, memory, disk. They install Prometheus and add a Grafana dashboard. Things break in production and they cannot understand why from metrics alone, so they add centralised logging with Splunk or ELK. A new service gets an APM agent. Someone adds PagerDuty. Now every incident generates five concurrent notifications across three tools, none of which tell the engineer where to look first.

The fix is not removing tools. It is assigning each tool a lane and enforcing it.

The observability layer model

LayerToolOwnsDoes NOT own
MetricsPrometheus + GrafanaInfrastructure health, K8s resource utilisation, SLO burn rate, custom business metricsLog-derived insights, distributed traces
LogsSplunk / LokiStructured log search, error pattern detection, compliance audit trailTime-series metrics, service topology
TracesJaeger / TempoRequest lifecycle, latency breakdown, cross-service dependenciesInfrastructure metrics, raw log storage
AlertingAlertmanager + PagerDutyRouting, deduplication, silencing, escalationAlert definition — that belongs to each source tool

The critical rule: each tool alerts only on what it uniquely knows. Prometheus does not fire alerts on log error patterns — that is Splunk's job. Splunk does not alert on pod memory pressure — that is Prometheus. The moment a signal appears in two alerting systems simultaneously, you have duplication that will cause alert fatigue.

Prometheus: metrics done right

kube-prometheus-stack as your foundation

Deploy via the kube-prometheus-stack Helm chart. It bundles Prometheus Operator, a curated set of recording rules, default Kubernetes dashboards, and Alertmanager. Do not start from scratch — the defaults are excellent and you can override what you need.

prometheus:
  prometheusSpec:
    retention: 30d
    retentionSize: 50GB
    storageSpec:
      volumeClaimTemplate:
        spec:
          storageClassName: fast-ssd
          resources:
            requests:
              storage: 100Gi

alertmanager:
  config:
    route:
      group_by: ['alertname', 'cluster', 'service']
      group_wait: 30s
      repeat_interval: 12h
      routes:
        - match: { severity: critical }
          receiver: pagerduty-critical
        - match: { severity: warning }
          receiver: slack-warning

Alert rules that do not cry wolf

The most common Prometheus anti-pattern is threshold alerts: "alert if CPU above 80%." These fire constantly on healthy systems doing real work and get ignored. Replace them with rate-of-change alerts, sustained-condition alerts, and SLO burn rate alerts.

groups:
  - name: slo-burn-rate
    rules:
      - alert: HighErrorBudgetBurnRate
        expr: |
          (
            rate(http_requests_total{status=~"5.."}[1h])
            / rate(http_requests_total[1h])
          ) > (5 * 0.001)
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "High error budget burn on {{ $labels.service }}"
          runbook: "https://runbooks.example.com/high-error-rate"

      - alert: PodMemoryPressureSustained
        expr: |
          container_memory_working_set_bytes
          / container_spec_memory_limit_bytes > 0.9
        for: 15m    # sustained 15 min, not a spike
        labels:
          severity: warning

Custom business metrics via exporters

Infrastructure metrics tell you the platform is running. Business metrics tell you it is doing useful work. Build lightweight exporters that scrape your application's internal APIs and expose metrics in Prometheus format:

from prometheus_client import start_http_server, Gauge, Counter
import requests, time

QUEUE_DEPTH     = Gauge('job_queue_depth', 'Jobs waiting', ['queue_name'])
PROCESSED_TOTAL = Counter('jobs_processed_total', 'Jobs completed', ['status'])
P99_LATENCY     = Gauge('job_processing_seconds_p99', 'P99 latency')

def collect():
    stats = requests.get('http://app/internal/stats').json()
    for q in stats['queues']:
        QUEUE_DEPTH.labels(queue_name=q['name']).set(q['depth'])
    P99_LATENCY.set(stats['latency_p99'])

if __name__ == '__main__':
    start_http_server(9100)
    while True:
        collect()
        time.sleep(30)

Grafana: dashboards engineers actually open

Dashboard-as-code with provisioning

Store all dashboards as JSON in Git. Grafana's sidecar provisioner loads them automatically on startup — no manual import, no dashboards that only exist on one instance. Changes go through PR review like any other code change.

grafana:
  sidecar:
    dashboards:
      enabled: true
      label: grafana_dashboard
      searchNamespace: ALL

---
apiVersion: v1
kind: ConfigMap
metadata:
  name: my-service-dashboard
  labels:
    grafana_dashboard: "1"
data:
  my-service.json: |
    { ... dashboard JSON ... }

USE and RED as your panel template

Every service dashboard should answer two sets of questions. For infrastructure resources (USE): Utilisation, Saturation, Errors. For user-facing services (RED): Rate, Errors, Duration. A dashboard that answers these six questions clearly for each critical service is more valuable than a 40-panel dashboard that answers none of them decisively.

Splunk: logs as a first-class signal

Index strategy by retention, not by application

[application_logs]
frozenTimePeriodInSecs = 7776000     # 90 days

[audit_logs]
frozenTimePeriodInSecs = 31536000    # 1 year
coldToFrozenDir = /archive/audit

[debug_logs]
frozenTimePeriodInSecs = 604800      # 7 days — no reason to keep debug logs longer

Structured logging: the ops contract

When every log line has consistent fields — level, service, trace_id, duration_ms, error_code — Splunk searches become trivial and dashboards build themselves. Raw text logs are a symptom of an application that was not designed to be operated.

# Unstructured — painful to query:
index=app_logs "ERROR" | rex "user (?<user_id>\d+) failed"

# Structured JSON — instant:
index=app_logs level=ERROR | stats count by service, error_code

Enforce structured logging at the framework level. Add a logging middleware or base logger to your internal libraries so teams get structured logs by default, not as an afterthought.

Alertmanager: the routing brain

Alertmanager routes, deduplicates, and silences — it does not define alerts. Keep alert rules in Prometheus. Use Alertmanager exclusively for deciding who gets paged, when, and how. The routing tree that works in practice:

  • Critical + production → PagerDuty, immediate page
  • Warning + production → Slack ops channel, no page
  • Any + non-production → Slack dev channel, no page, no escalation
  • Informational → logged to summary index, no notification

Inhibition rules are equally important. When a node goes down, suppress the cascade of pod-level alerts that fire as a consequence — they are not independent problems.

inhibit_rules:
  - source_match:
      alertname: NodeDown
    target_match_re:
      alertname: (PodNotReady|PodCrashLooping|DeploymentReplicasMismatch)
    equal: ['node']

The result

After restructuring observability along these lines on a large cloud platform: alert volume dropped roughly 60%. Mean time to acknowledge dropped because each alert arrived with a clear owner and a runbook link. Mean time to diagnose dropped because engineers knew exactly which tool to open for each alert type. And post-incident reviews shifted from "why did we not know about this?" to "here is exactly what happened and when."

Observability is not a tool purchase. It is an engineering discipline. The tools are just the implementation.
