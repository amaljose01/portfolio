GitOps in practice: structuring ArgoCD for multi-environment deployments

Amal Jose 2025 8 min read Cloud Operations & Automation Engineer

GitOps sounds simple on paper: Git is the source of truth, and your cluster converges to whatever is in the repo. In practice, most teams get about three environments in before the repository structure collapses under its own weight — duplicated manifests, unclear promotion paths, and drift that nobody notices until production looks nothing like staging.

I've set up ArgoCD-based GitOps across several production environments, and this post documents the patterns that held up: how to structure your repo, how to use ApplicationSets properly, how to handle secrets, and how to design a promotion flow that's both safe and fast.

The repo structure problem

The most common GitOps mistake is treating your repository like a folder of kubectl apply files. It works for one environment. Add a second environment and you're already copy-pasting. Add a third and you have a synchronisation problem masquerading as a deployment problem.

The structure that works well at scale separates three concerns:

  • Application definitions — what you're deploying (Helm charts, base manifests)
  • Environment configuration — how it's configured per environment (values overrides)
  • Cluster configuration — what ArgoCD applications exist on which cluster
gitops-repo/
├── apps/
│   ├── my-service/
│   │   ├── Chart.yaml
│   │   ├── templates/
│   │   ├── values.yaml           # shared defaults
│   │   ├── values-dev.yaml
│   │   ├── values-staging.yaml
│   │   └── values-prod.yaml
│   └── platform-components/
│       ├── cert-manager/
│       ├── external-secrets/
│       └── ingress-nginx/
├── clusters/
│   ├── dev/
│   │   └── apps.yaml             # ArgoCD App-of-Apps for dev
│   ├── staging/
│   │   └── apps.yaml
│   └── prod/
│       └── apps.yaml
└── argocd/
    ├── projects/
    └── applicationsets/

The clusters/ directory is the entry point for ArgoCD. Each cluster has a single App-of-Apps (or ApplicationSet) that bootstraps everything else. New environments are added by creating a new directory here, not by duplicating anything in apps/.

ApplicationSet: one template, all environments

ApplicationSet is ArgoCD's multiplier. Instead of creating individual Application objects per environment, you define a template once and generate applications dynamically. This is the pattern I use for services that need to run across dev, staging, and prod:

apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: my-service
  namespace: argocd
spec:
  generators:
    - list:
        elements:
          - env: dev
            cluster: https://dev-cluster.example.com
            namespace: my-service-dev
            autosync: "true"
          - env: staging
            cluster: https://staging-cluster.example.com
            namespace: my-service-staging
            autosync: "true"
          - env: prod
            cluster: https://prod-cluster.example.com
            namespace: my-service-prod
            autosync: "false"    # prod is manual sync only
  template:
    metadata:
      name: 'my-service-{{env}}'
      annotations:
        notifications.argoproj.io/subscribe.on-sync-failed.slack: ops-alerts
    spec:
      project: default
      source:
        repoURL: https://github.com/your-org/gitops-repo
        targetRevision: main
        path: apps/my-service
        helm:
          valueFiles:
            - values.yaml
            - 'values-{{env}}.yaml'
      destination:
        server: '{{cluster}}'
        namespace: '{{namespace}}'
      syncPolicy:
        automated:
          prune: true
          selfHeal: '{{autosync}}'
        syncOptions:
          - CreateNamespace=true
          - ServerSideApply=true

Production sync policy: Always set autosync: false for production. Automated self-healing on prod means a mistyped config in Git gets applied instantly. Manual sync keeps a human in the loop for the last mile.

Sync waves: ordering your deployments

When multiple resources deploy together, order matters. CRDs before controllers. ConfigMaps before Deployments. Databases before applications. ArgoCD sync waves handle this with a single annotation:

# Wave 0: infrastructure (CRDs, namespaces, RBAC)
metadata:
  annotations:
    argocd.argoproj.io/sync-wave: "0"

# Wave 1: config and secrets
metadata:
  annotations:
    argocd.argoproj.io/sync-wave: "1"

# Wave 2: databases and stateful services
metadata:
  annotations:
    argocd.argoproj.io/sync-wave: "2"

# Wave 3: application workloads
metadata:
  annotations:
    argocd.argoproj.io/sync-wave: "3"

# Wave 4: ingress and external routing
metadata:
  annotations:
    argocd.argoproj.io/sync-wave: "4"

Resources in wave 0 must be healthy before wave 1 starts. This eliminates an entire category of "it worked the second time" failures caused by resource creation race conditions.

Secrets: what never goes in Git

The GitOps security rule is absolute: no secrets in Git, ever, even encrypted. The two patterns I've used in production:

External Secrets Operator + Vault

ExternalSecret manifests live in Git. They contain only a reference to where the secret lives in Vault — not the value. The External Secrets Operator reads the reference and creates a native Kubernetes Secret at runtime.

apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: database-credentials
  namespace: my-service-prod
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: vault-backend
    kind: ClusterSecretStore
  target:
    name: database-credentials
    creationPolicy: Owner
  data:
    - secretKey: DB_PASSWORD
      remoteRef:
        key: my-service/prod/database
        property: password
    - secretKey: DB_USERNAME
      remoteRef:
        key: my-service/prod/database
        property: username

Sealed Secrets (for teams without Vault)

Sealed Secrets encrypts a secret with a cluster-specific public key. The encrypted SealedSecret is safe to commit — only that specific cluster can decrypt it. Simpler to set up than Vault, but less flexible for rotation and cross-cluster sharing.

Promotion flow: dev → staging → prod

A GitOps promotion flow should be explicit, reviewable, and produce an audit trail. Here's the pattern:

  1. Dev auto-deploys on every merge to main — ArgoCD picks up the commit and syncs within 3 minutes.
  2. Staging promotion is triggered by updating values-staging.yaml with the new image tag. This happens via a CI pipeline step that opens a PR. A team member reviews and merges.
  3. Production promotion follows the same PR process for values-prod.yaml, with a required approval from a senior engineer or release manager. After merge, ArgoCD shows the app OutOfSync — a human triggers the sync.
# CI pipeline step: create promotion PR (GitHub Actions example)
- name: Promote to staging
  run: |
    git checkout -b promote/my-service-${{ env.IMAGE_TAG }}
    
    # Update image tag in staging values
    yq e '.image.tag = "${{ env.IMAGE_TAG }}"' \
      -i apps/my-service/values-staging.yaml
    
    git add apps/my-service/values-staging.yaml
    git commit -m "chore: promote my-service ${{ env.IMAGE_TAG }} to staging"
    git push origin promote/my-service-${{ env.IMAGE_TAG }}
    
    gh pr create \
      --title "Promote my-service ${{ env.IMAGE_TAG }} to staging" \
      --body "Automated promotion from dev. Review diff before merging." \
      --base main

Every promotion is a pull request. Every production deployment is a commit in Git. The full deployment history is a git log away. When an incident happens at 2am, you're not wondering what changed — you're reading the PR that caused it.

Drift detection and self-healing

One of GitOps's most underrated operational benefits is drift detection. ArgoCD continuously compares the live cluster state against the declared state in Git. When they diverge — someone ran a one-off kubectl edit during an incident, a node replacement changed a resource — ArgoCD surfaces it immediately.

For non-production clusters, selfHeal: true automatically reconciles drift. For production, I prefer drift to surface as an alert rather than an automatic revert — an emergency manual change might be intentional, and you want to decide deliberately whether to revert it or codify it in Git.

Common pitfalls

  • Too many applications, not enough projects. ArgoCD projects are your RBAC and network policy boundary. Structure projects around teams or delivery domains, not individual services.
  • Ignoring health checks. ArgoCD's sync status (Synced/OutOfSync) and health status (Healthy/Degraded) are different things. A deployment can be Synced but Degraded if the pods aren't running. Always watch both.
  • Not setting resource limits. ArgoCD will faithfully apply whatever is in your manifests — including manifests with no resource limits. Set limits in your Helm chart defaults and enforce them via OPA/Kyverno policies.
  • Forgetting about CRD upgrades. When a Helm chart bumps a CRD version, ArgoCD won't automatically upgrade existing CRDs — it's a known limitation. Handle CRD upgrades explicitly in your release process.

Is it worth it?

Every team I've introduced GitOps to has asked this question during the first week, usually while debugging a sync wave issue or a resource ordering problem. The answer becomes clear around week three, when a junior engineer confidently deploys to production because the process is just "open a PR," and when a post-incident review traces the exact config change that caused a problem to a specific commit with a specific author and a specific review approval.

GitOps doesn't eliminate complexity. It makes complexity legible. That's the whole value.
